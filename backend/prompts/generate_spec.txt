## 1. Problem Framing

### Core Technical Challenges

Identify the 2-3 HARDEST problems THIS specific system must solve.
Not generic problems like "scaling" or "performance."
Specific problems derived from the user's idea and constraints.

**CRITICAL**: Core challenges must be the genuinely hard distributed systems problems specific to this idea — not generic challenges.

To identify the correct core challenges, ask these questions about the system being designed:

1. **What is the primary real-time coordination problem?**
   (matching, routing, ordering, synchronization, fan-out)

2. **What is the primary consistency problem?**
   (state machines with failure modes, double-booking, race conditions, distributed transactions)

3. **What is the primary data locality problem?**
   (geospatial search, proximity, sharding by location, hot partitions)

4. **What is the primary scale asymmetry?**
   (celebrity/hot user problem, read-heavy vs write-heavy imbalance, bursty vs steady traffic)

5. **What is the primary media or content challenge?**
   (large blob storage, transcoding, CDN invalidation, streaming vs download)

For each system, identify which 2-3 of these questions produce the genuinely hard answer for this specific idea. Those are the core challenges.

**Special case for workflow systems**: If the system has orders, bookings, transactions, or any multi-step workflow — the order state machine and its failure modes is always a core challenge. Specifically: what happens when an actor in the workflow goes offline mid-process, rejects their step, or takes too long. This is never solved by CDN or file upload patterns.

**Never identify standard CRUD, basic auth, or file upload as core challenges** — these are solved problems regardless of domain.

**For location-based systems where proximity search is required, always address geospatial indexing specifically** — VARCHAR location columns are never acceptable for systems that need nearby search.

### Actual Schema

```sql
-- PostgreSQL Schema
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    created_at TIMESTAMP DEFAULT NOW(),
    last_login TIMESTAMP,
    INDEX idx_email (email),
    INDEX idx_last_login (last_login DESC)
);

CREATE TABLE sessions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token VARCHAR(255) UNIQUE NOT NULL,
    expires_at TIMESTAMP NOT NULL,
    created_at TIMESTAMP DEFAULT NOW(),
    INDEX idx_token (token),
    INDEX idx_user_expires (user_id, expires_at)
);

CREATE TABLE resources (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    name VARCHAR(500) NOT NULL,
    metadata JSONB,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW(),
    INDEX idx_user_created (user_id, created_at DESC),
    INDEX idx_metadata_gin (metadata) USING gin
);

CREATE TABLE events (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    resource_id UUID NOT NULL REFERENCES resources(id) ON DELETE CASCADE,
    event_type VARCHAR(100) NOT NULL,
    payload JSONB NOT NULL,
    occurred_at TIMESTAMP DEFAULT NOW(),
    INDEX idx_resource_occurred (resource_id, occurred_at DESC),
    INDEX idx_type_occurred (event_type, occurred_at DESC)
);
```
